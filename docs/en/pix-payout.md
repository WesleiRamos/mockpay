# PIX Payout

PIX Payouts represent withdrawal requests (PIX Send). Each payout is created with `PROCESSING` status and auto-liquidates after 30 seconds, simulating the PIX transfer flow. A webhook is dispatched when liquidation completes.

## Create a PIX Payout

```
POST /v1/pix/payouts
```

```json
{
  "amount": 5000,
  "pix_key_type": "cpf",
  "pix_key": "12345678901",
  "external_id": "019b38f7-2b67-7e0a-a8f3-5c4f72b61f1d",
  "idempotency_key": "pix-withdrawal:payout:019b38f7-2b67-7e0a-a8f3-5c4f72b61f1d",
  "metadata": {
    "source": "wallet"
  }
}
```

### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `amount` | int | yes | Amount in cents (must be > 0) |
| `pix_key_type` | string | yes | PIX key type: `cpf`, `phone`, `email`, or `random` |
| `pix_key` | string | yes | Recipient's PIX key |
| `external_id` | string | no | Your own identifier for this payout |
| `idempotency_key` | string | yes | Idempotency key generated by the caller |
| `metadata` | object | no | Key-value pairs for additional data |

### Response

```json
{
  "data": {
    "id": "payout_01HXZ6R0V7V8D8QN7M5W6B8J9K",

```



## Status Lifecycle

```
PROCESSING → LIQUIDATED   (automatic, after 30 seconds)
```

Wallets should interpret the following statuses:
- `PROCESSING` — payout sent, awaiting liquidation
- `LIQUIDATED` or `PAID` — payout successfully liquidated
- `FAILED`, `REFUSED`, `REJECTED` or `CANCELLED` — payout failed, the wallet releases the reserve

## Auto Liquidation

A background job runs every 30 seconds to check for pending payouts:

1. Lists all payouts with status `PROCESSING`
2. Checks if 30 seconds have passed since creation
3. If eligible, sets status to `LIQUIDATED`
4. Generates an `end_to_end_id` (format: `E20260502...`)
5. Dispatches `pix.payout.liquidated` webhook

## Idempotency

- `POST /v1/pix/payouts` with the same `idempotency_key` returns the same payout
- `GET /v1/pix/payouts/{payout_id}/check` returns the current state without creating a new payout
- Repeated webhooks with the same `id` can be re-sent; the wallet deduplicates the event
- The `amount` returned in the payout must match the requested `amount`, otherwise the wallet rejects the liquidation

## Webhook

When the payout is liquidated, MockPay sends a `POST` to the configured `MOCKPAY_WEBHOOK_URL`:

```json
{
  "id": "evt_payout_01HXZ6R0V7V8D8QN7M5W6B8J9K",
  "type": "pix.payout.liquidated",
  "payload": {
    "id": "payout_01HXZ6R0V7V8D8QN7M5W6B8J9K",
    "amount": 5000,
    "status": "LIQUIDATED",
    "external_id": "019b38f7-2b67-7e0a-a8f3-5c4f72b61f1d",
    "end_to_end_id": "E1823612020260503120000000000001",
    "metadata": {
      "source": "wallet"
    },
    "created_at": "2026-05-03T12:00:00.000",
    "updated_at": "2026-05-03T12:01:00.000"
  },
  "created_at": "2026-05-03T12:01:00.000"
}
```

The `X-MockPay-Signature` header is included with HMAC-SHA256 when `MOCKPAY_WEBHOOK_SECRET` is configured.

## cURL Examples

```bash
# Create a payout
curl -X POST "http://localhost:8080/v1/pix/payouts" \
  -H "Authorization: Bearer mock_key" \
  -H "Content-Type: application/json" \
  -d '{
    "amount": 5000,
    "pix_key_type": "cpf",
    "pix_key": "12345678901",
    "external_id": "019b38f7-2b67-7e0a-a8f3-5c4f72b61f1d",
    "idempotency_key": "pix-withdrawal:payout:019b38f7-2b67-7e0a-a8f3-5c4f72b61f1d",
    "metadata": {
      "source": "wallet"
    }
  }'

# Check payout status
curl -X GET "http://localhost:8080/v1/pix/payouts/payout_01HXZ6R0V7V8D8QN7M5W6B8J9K/check" \
  -H "Authorization: Bearer mock_key"
```
